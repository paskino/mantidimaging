mantidimaging.core.parallel.two_shared_mem.execute(data=None, second_data=None, partial_func=None, cores=None, chunksize=None, name='Progress', progress=None)[source]¶

Executes a function in parallel with shared memory between the processes.

The array must have been created using parallel.utility.create_shared_array(shape, dtype).

If the input array IS NOT a shared array, the data will NOT BE CHANGED!

The reason for that is that the processes don’t work on the data, but on a copy.

When they process it and return the result, THE RESULT IS NOT ASSIGNED BACK TO REPLACE THE ORIGINAL as is done in parallel.exclusive_mem, it is merely discarded.

  • imap_unordered gives the images back in random order

  • map and map_async do not improve speed performance

  • imap seems to be the best choice

Using _ in the for _ enumerate is slightly faster, because the tuple from enumerate isn’t unpacked, and thus some time is saved.

From performance tests, the chunksize doesn’t seem to make much of a difference, but having larger chunks usually led to slower performance:

Shape: (50,512,512) 1 chunk 3.06s 2 chunks 3.05s 3 chunks 3.07s 4 chunks 3.06s 5 chunks 3.16s 6 chunks 3.06s 7 chunks 3.058s 8 chunks 3.25s 9 chunks 3.45s

Parameters

  • data – the shared data array that will be processed in parallel

  • second_data – the second shared data array that will be processed in parallel

  • partial_func – a function constructed using partial to pass the correct arguments

  • cores – number of cores that the processing will use

  • chunksize – chunk of work per process(worker)

  • name – name of the task used in progress reporting

  • progress – Progress instance to use for progress reporting (optional)

mantidimaging.core.parallel.two_shared_mem.inplace(func, i, **kwargs)[source]¶
Use if the parameter function will do the following:

  • Perform an operation on the input data that is dependent on another container

  • DOES NOT have a return statement

  • The data is NOT RESIZED

  • The data will be changed INPLACE inside the function being forwarded (the parameter function)

You HAVE to be careful when using this, for example the func:

def _apply_normalise_inplace(

data, dark=None, norm_divide=None, clip_min=None, clip_max=None):

data = np.clip(np.true_divide(

data - dark, norm_divide), clip_min, clip_max)

DOES NOT CHANGE THE DATA! Because the data = … variable inside is just a LOCAL VARIABLE that is discarded.

The proper way to write this function is: def _apply_normalise_inplace(

data, dark=None, norm_divide=None, clip_min=None, clip_max=None):

data[:] = np.clip(np.true_divide(

data - dark, norm_divide), clip_min, clip_max)

Notice the data[:], what this does is REFER to the ACTUAL parameter, and then changes it’s contents, as [:] gives a reference back to the inner contents.

Parameters

  • func – Function that will be executed

  • i – index from the shared_data on which to operate

  • kwargs – kwargs to forward to the function func that will be executed

Returns

nothing is returned, as the data is replaced in place

  • Perform an operation on the input data that is dependent on the same second parameter
